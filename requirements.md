# 时间感知工具 — 需求文档

> 本文档由 office-hours + plan-ceo-review 流程产出。
> 状态：已锁定 MVP 范围，自动重排功能保留。

---

## 1. 产品概述

一个**有时间感知能力的个人日程工具**，帮助用户看见一天的时间去向，并在现实打断计划时自动调整后续安排。

核心理念：**不是静态日历，而是动态时间调度器。**

---

## 2. 目标用户

**核心用户：** 自主管理时间的知识工作者——开发者、设计师、写作者、自由职业者、产品经理。

**要完成的任务：** "我想知道时间花在了哪里，并且在现实变化时，相信我的日程会自动适应。"

---

## 3. 痛点（来自用户真实场景）

1. **打断后的时间黑洞** —— 突然有事情进来，原来的任务停下，不知道已经做了多久、还剩多久。
2. **只有开始提醒，没有结束提醒** —— 日历只催开始，不催结束，导致任务超时、连续作战、没有休息。
3. **计划 vs 实际脱节** —— 事后回想"今天到底干了什么"，现有工具答不上来。
4. **时间感知缺失** —— 一天结束不知道自己时间去哪了，未规划的时间（上厕所、刷手机）完全不可见。

## 4. MVP 范围（v1）

### 4.1 核心功能（必须有）

| 功能 | 说明 | 优先级 |
|------|------|--------|
| **快速记录** | 打开网页，点击"开始"→输入任务名→计时启动。点击"结束"→记录耗时。不需要提前规划，随时可记。 | P0 |
| **计划任务** | 预先安排明天的任务和预计时长，生成时间线。到时间可选"开始执行"或"跳过"。 | P0 |
| **时间线视图** | 一天 24 小时纵向排列，每个任务以色块展示实际占用的时间段。 | P0 |
| **打断处理** | 任务 A 进行中，点击"有事插入"→记录新任务。原任务暂停，可选择：恢复继续 / 延后 / 取消。 | P0 |
| **自动重排** | 打断结束后，后续计划任务自动后推，展示新时间线。用户可一键接受或手动调整。 | P0 |
| **结束提醒** | 计划任务快结束时弹窗提示："时间到了，结束/延长/延后？" | P0 |
| **未记录时间高亮** | 时间线上空白的时间段用特殊颜色标出，提醒"这段时间没记录"。 | P1 |
| **今日报告** | 一天结束后看统计：计划内多久、临时记录多久、空白多久、被打断几次、计划被调整几次。 | P1 |

### 4.2 体验优化（MVP 尽量带）

| 功能 | 说明 |
|------|------|
| **快捷预设** | 常见临时任务一键开始（上厕所、喝水、刷手机、休息），不用打字。 |
| **事后补录** | 发现刚才忘了记，直接在时间线空白处点击，回填一段时间。 |
| **本地存储** | 数据存在浏览器本地，刷新不丢，关掉电脑明天打开还在。每 30 秒自动保存。 |
| **任务预估时长** | 每个任务（包括临时任务）必须有预估时长，用于重排算法。提供快捷选项：5分钟、15分钟、30分钟、1小时、不确定。 |

### 4.3 不做（v2 及以后）

- 日历同步（Google/Outlook）
- 周/月视图
- 数据导出
- 多设备同步
- 账号登录
- 团队协作
- AI 建议
- 复杂分类标签

## 5. 用户故事

### 故事 A：正常执行
> 作为用户，我早上打开应用，看到今天的计划时间线。我点击"开始"第一个任务，计时启动。倒计时显示剩余计划时间。结束前 5 分钟，应用提示我"该收尾了"。计划时间到时，弹窗问我"结束、延长、还是延后？"我点击"结束"，任务标记完成，应用推荐我下一个任务。

### 故事 B：打断处理
> 作为用户，我正在执行任务 A（计划 30 分钟，已进行 10 分钟）。突然有事，我点击"有事插入"。应用问我"发生了什么？"我输入"紧急 bug"。应用提供选项：暂停+开始新任务 / 替换原任务 / 快速记录+返回。我选择"暂停+开始新任务"。任务 B 开始运行，任务 A 显示"已做 10 分钟，剩余 20 分钟"。任务 B 结束后，应用问我"恢复任务 A？"我点击是，应用自动重新计算今日剩余时间线。

### 故事 C：自动重排
> 作为用户，打断消耗了 45 分钟，任务 A 还剩 20 分钟。应用计算后续安排：把所有后续任务后推 45 分钟？还是砍掉低优先级？应用展示两个选项："保留全部"（日程延长）或"砍掉缓冲"（删除休息/低优先级任务）。我选择一个，时间线更新，视觉差异显示哪些变了。

### 故事 D：临时记录
> 作为用户，我没做任何计划，只是打开应用点击"开始"，记录"刷手机"。30 分钟后我点击结束。这个色块出现在时间线上。晚上我看报告，发现"临时记录 2.5 小时，空白 3 小时"。

### 故事 E：每日复盘
> 作为用户，一天结束我打开"今日报告"。看到时间线对比（计划蓝 vs 实际绿 vs 打断红）、统计数据（计划 7h，实际工作 4.5h，打断 1.5h，休息 1h）、洞察（"3 项任务按时完成。1 项延后到明天。今天计划被调整了 4 次。"）、主要打断来源（"Slack 3 次 45 分钟"）。

## 6. 数据模型

```typescript
// 一件任务
type Task = {
  id: string;
  title: string;
  plannedDurationMinutes: number;
  actualDurationMinutes?: number;
  status: 'scheduled' | 'active' | 'paused' | 'completed' | 'cancelled' | 'deferred';
  scheduledStart?: Date;
  scheduledEnd?: Date;
  originalScheduledStart?: Date;   // 首次排程，用于复盘对比
  originalScheduledEnd?: Date;
  actualStart?: Date;
  actualEnd?: Date;
  category?: string;
  priority: 'low' | 'medium' | 'high';
  revisionCount: number;           // 被重排了几次
};

// 打断事件
type Interruption = {
  id: string;
  timestamp: Date;
  interruptedTaskId: string;
  newTaskId?: string;
  description: string;
  durationMinutes?: number;
};

// 一天的日程
type DaySchedule = {
  date: string;                    // YYYY-MM-DD
  tasks: Task[];
  interruptions: Interruption[];
};

// 历史洞察
type TimeInsight = {
  date: string;
  totalPlannedMinutes: number;
  totalActualMinutes: number;
  totalInterruptionMinutes: number;
  completedTasks: number;
  cancelledTasks: number;
  deferredTasks: number;
  categoryBreakdown: Record<string, number>;
  revisionCount: number;
};
```

## 7. 核心交互流程

### 7.1 状态机（打断处理）

```
[运行中] --打断--> [暂停] --插入新任务--> [新任务运行中]
   |                      |
   |                      +--恢复--> [运行中] + 触发重排
   +--结束--> [完成]
```

### 7.2 重排算法阴影路径

| 场景 | 处理 |
|------|------|
| 新任务无预估时长 | 提示"请输入预计时长才能调整后续安排"，不执行重排 |
| 后推导致跨天 | 提示"XX 任务会排到明天，是否删除/延后/缩短？" |
| 用户拒绝新排程 | 时间线保持原样，新任务作为"未安排"悬浮，报告标注"计划冲突" |
| 原任务未开始被打断 | 无"已用时间"，恢复时从零开始 |
| 浏览器崩溃恢复 | 读取 localStorage 备份，询问"是否继续上次的任务？" |

## 8. 界面要求

### 8.1 核心原则

- **默认单天视图。** 周视图是次要的。
- **时间线，不是网格。** 纵向排列的时间段，不是日历方格。
- **极简界面。** 这一天本身就是界面。
- **移动优先。** 个人工具，工作时用手机。

### 8.2 颜色状态

| 状态 | 颜色 |
|------|------|
| 计划中 | 蓝色 |
| 进行中/正常 | 绿色 |
| 超时 | 黄色 |
| 被打断 | 红色 |
| 完成/取消 | 灰色 |
| 未记录 | 透明/浅色高亮 |

### 8.3 首页层次

1. **巨大的"开始"按钮** —— 降低行动门槛
2. **今日时间线缩略** —— 一眼看见已记录内容
3. **未记录时间百分比** —— 温和紧迫感
4. **快捷预设** —— 上厕所、喝水、刷手机、深度工作，一键开始

### 8.4 空状态

- **首次使用：** "你的今天是什么样的？"→ 直接拖第一个任务块
- **无计划日：** "今天没有计划，随时点击开始记录"
- **全部空白：** "0% 已记录" + "点击按钮，看见你的时间"

## 9. 技术架构（MVP）

| 层级 | 技术 |
|------|------|
| 前端框架 | React + TypeScript |
| 样式 | Tailwind CSS |
| 状态管理 | Zustand |
| 存储 | LocalStorage / IndexedDB（本地优先） |
| 计时器 | Web Worker（后台计时准确，不用 setInterval 累加） |
| 通知 | Web Notifications API |
| 打包 | Vite |
| 部署 | PWA（可安装、离线可用） |

**为什么不做后端？**
- 本质是单人工具
- 降低演示阶段复杂度
- 时间数据涉及隐私
- 验证后再加云端同步

## 11. 风险与假设

| 风险 | 缓解 |
|------|------|
| 用户忘记点开始/结束 | 首页高亮未记录时间；超过计划时长 2 倍时弹窗确认 |
| 浏览器刷新丢数据 | 每 30 秒自动写入 localStorage |
| 重排算法让人焦虑 | 语气温柔，不是"日程已崩溃"，而是"帮你调整了，看看这样行吗？" |
| 临时任务输入负担 | 快捷预设 + 预估时长默认值 |
| 移动端电池 drain | 轻量 Web Worker，无持续动画 |

**最大假设：** 用户愿意在每次任务切换时主动点击"开始/结束"。
**验证方式：** 自用 3 天，看是否每天能点击 5 次以上。

## 12. 开发顺序

1. 底层数据模型 + LocalStorage

4. 快速记录流程
5. 打断处理 + 重排算法
6. 计划任务 builder
7. 今日报告
