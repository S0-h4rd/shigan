## Section 1: 架构审查

### 1.1 状态管理设计

**Store 结构：**

```typescript
interface AppState {
  // 数据层
  schedule: DaySchedule;

  // 运行时状态（全局唯一）
  activeTaskId: string | null;
  pausedTaskId: string | null;    // 被打断时暂停的原任务

  // 计时锚点
  timerStartAt: number | null;    // Date.now()，非累加值

  // UI
  view: 'timeline' | 'report';
}
```

**关键原则：**
- `actualDuration` 不通过 `setInterval` 累加，每次读取时计算 `Date.now() - actualStart`
- 运行时状态与任务数据状态分离：`Task.status` 存业务状态（scheduled/completed/cancelled），`activeTaskId` 存运行时状态

### 1.2 计时器引擎

| 方案 | 精度 | 后台运行 | 复杂度 | 结论 |
|------|------|---------|--------|------|
| `setInterval` | 中 | 被节流 | 低 | ❌ |
| Web Worker | 高 | 正常 | 中 | ❌ |
| `Date.now()` 差值 | 高 | 无关 | 低 | ✅ |

**采用 `Date.now()` 差值方案。** 不需要 Web Worker。每次显示剩余时间时现场计算 `plannedEnd - Date.now()`。后台标签页被节流不影响精度，切回来一算是准的。

### 1.3 自动重排算法

**位置：** `core/reschedule.ts` 纯函数模块

```typescript
export function reschedule(
  tasks: Task[],
  interruptedTaskId: string,
  newTask: Task
): Task[];
```

**原则：**
- 纯函数：输入旧任务列表 + 打断信息，输出新任务列表
- 不涉及 UI、不涉及存储
- 可单元测试、可复用

### 1.4 数据持久化策略

- **存储介质：** LocalStorage
- **保存内容：** 整个 `DaySchedule` + `timerStartAt` + `activeTaskId`
- **保存频率：** 每 30 秒自动保存 + 关键操作后立即保存（开始/结束/打断）
- **恢复策略：** 启动时读取 LocalStorage，若存在 `activeTaskId` 且 `timerStartAt` 存在，弹窗询问"继续上次的任务？"

**风险缓解：** 只存当天数据。历史数据归档到单独的 `history-YYYY-MM.json` 文件，避免 LocalStorage 溢出。

### 1.5 打断模型

**采用扁平化设计。** 任何时候最多只有一个 active task，不支持嵌套打断栈。

```
状态流转：
  idle → active（开始任务）
  active → paused（打断，暂停当前）
  paused → active（恢复）
  active → idle（结束任务）
```

## Section 2: 代码质量审查

### 2.1 类型安全

**运行时状态与数据状态分离：**

```typescript
// 业务状态（持久化）
type TaskStatus = 'scheduled' | 'completed' | 'cancelled' | 'deferred';

// 运行时状态（内存）
type RuntimeStatus = 'idle' | 'active' | 'paused';  // 全局只有一个
```

禁止给每个 Task 配独立的 runtime status，避免竞态和状态不一致。

### 2.2 错误处理

| 场景 | 处理方式 |
------|---------|
| LocalStorage 满（QuotaExceededError） | 捕获异常，提示用户清理历史数据 |
| LocalStorage 被禁用 | 降级为内存存储，提示"数据不会持久化" |
| 无效的任务 ID | `throw` + React Error Boundary 捕获 |
| 重排算法输入异常 | 纯函数返回 `null`，调用方回退到原 schedule |
| 跨天任务重排 | 边界检查，超出当天范围时截断或标记 |

### 3.2 单元测试（高优先级）

| 模块 | 测试函数 | 测试场景 |

- 刷新页面 → 恢复进行中的任务（持久化恢复流）

## Section 4: 性能审查

### 4.1 LocalStorage 写入

**风险：** 每 30 秒写入，若数据量大可能卡顿。
**缓解：**
- 只保存当天 `DaySchedule`
- 历史数据按月归档到独立 key
- 写入前 `JSON.stringify`，控制对象体积

### 4.2 渲染优化

**风险：** 时间线 20+ 色块，频繁重渲染。
**缓解：**
- 时间线计算用 `useMemo` 缓存
- 单个色块用 `React.memo` 包裹
- 不需要 `setInterval` 驱动渲染，用 `requestAnimationFrame` 或事件驱动

### 4.3 计时器精度

采用 `Date.now()` 差值，无累加误差，无后台节流问题。
